# DataLens — Phase 3 Test Results

> Date: 2026-06-18. Author: James McGhee / Heathen Engineering.
> Scope: the Phase-3 compute substrate (DataLens Core Stream A3–A5 + C ABI) and the HATE simulation
> layer built on it, as proven end-to-end in Unity. Captures both the **native correctness suite**
> and the **measured demo performance**.
>
> Lineage: Phase 0 (`Phase0_2025111313*.csv` / `Phase0_2025111314*.csv`) was the original UE
> byte-stride store benchmark. Phase 3 is the mature parallel substrate — bit-packed columns, the
> Lens scheduler, the serialisable IR, the tick/DataView loop, the stable C ABI — plus HATE
> (attributes / effects / abilities / utility AI) running on top of it.
>
> These are recorded results compiled from the project's `Performance-Log.md` and master plan, not a
> fresh run captured by this document.

## 1. What Phase 3 covers

| Layer | State at this snapshot |
|-------|------------------------|
| **A2** columnar store | Range-narrowed widths, 64B cache-line-aligned columns, dense Validity/Locked bitmasks (O(1) bit-scan alloc), per-row Simulation LOD. |
| **A3** Systems + Lens | Branchless column kernels (arith, bitwise, scaled fused-multiply, response-curve, counter-based noise, argmax) + same/mixed-type predicates; dependency-level parallel scheduler, row-parallel within a level. Deterministic (parallel == serial). |
| **A4** update IR | Flat, pointer-free, serialisable op set (store-by-index); `IrProgram` serialise/deserialise (v2, curve fields). |
| **A5** scheduler + DataView | Tick/cadence scheduler (Simulation LOD as execution frequency); read-only snapshot `DataView` with parallel refresh. |
| **C ABI** | 76 `dl_*` symbols; opaque handles + bulk byte-span transfer; loaded by the Unity Foundation via P/Invoke. |
| **HATE** (on the substrate) | Attribute State (Base/Current), Instant + Duration effects (EndTick buff-density), granted Tags + Trigger gating, Abilities (cost/cooldown/requirement + tasks/cues + granting), and the full §8 utility-AI score→perturb→select pipeline. |

## 2. Native correctness suite

Engine-agnostic C++17 core, Catch2.

| Metric | Result |
|--------|--------|
| Total cases | **114** |
| Pass | 113 |
| Expected fail (xfail) | 1 — `DataLens::Serialize`/`Deserialize` round-trip, marked `[!mayfail]`, tracked for the persistence/delta phase (A6). |
| AddressSanitizer + UBSan | **Clean** |
| ThreadSanitizer | **Clean** |

**Suite breakdown (by area):**

| Area | What it proves |
|------|----------------|
| Store / schema | Create, pre-allocate, convert-to-schema, cell round-trip, range-narrowing helpers. |
| Validity / alloc | Bitmask liveness, O(1) `AllocRow`/`FreeRow`, capacity exhaustion, slot reuse — no validity/data overlap, no OOB. |
| Systems (kernels) | Conditional column transforms; cross-column ops; bitwise ops + bitmask predicates; scaled fused-multiply; mixed-type predicates. |
| Curve | `test_curve.cpp` — Linear/Power/Smoothstep/Threshold, normalise-clamp, invert, combine/accumulate, predicate gating, parallel == serial. |
| Noise | `test_noise.cpp` — bounded SplitMix64 fill + the `Score += Variance*Noise` perturb; reproducible, serial == chunked. |
| Argmax | `test_argmax.cpp` — K-column reduce to a Choice index, tie-to-lowest, minScore floor, parallel == serial. |
| Lens / scheduling | Parallel batch == sequential; dependency-level packing; long conflict chains stay ordered; cache-line alignment. |
| IR | Build/execute == direct; cross-column/predicate/LOD round-trip; serialise round-trip (v2 curve); bad-buffer rejection; out-of-range index safe no-op. |
| Schedule | Cadence, phase, LOD-tiered ticks, period clamp / empty no-op. |
| DataView | Materialise, snapshot-vs-mutation, live-only + row mapping, LOD band, RowData layout, parallel refresh == serial. |
| C ABI | All managed-visible entry points (run/lens/IR/view/tick) round-trip across the boundary. |

**Confidence note:** the parallel paths (Lens pool, view refresh, noise, argmax) are TSan-verified
race-free, and every parallel kernel is asserted **bit-identical to its single-threaded reference** —
so determinism is a tested property, not an assumption.

## 3. Measured demo performance

**Test rig:** Ryzen 9 5950X (16C / 32T), 128 GB, Arch Linux — a high-end box; discount vs a typical
target. Unity Play mode, native `.so` Release. Figures are smoothed whole-frame wall-clock (settled
over a few seconds), not isolated microbench. The recurring finding: the **substrate is a small
fraction of the frame; the single-threaded managed texture scatter (render) dominates** — an
app/renderer concern, not a DataLens/HATE limit.

### 3a. `DataLensSwarmDemo` — pure substrate swarm, 1,000,000 agents

A 6-op integrate/wrap IR program run every tick by the Lens; a positions `DataView` refreshed each
tick, bulk-copied and scattered into a texture.

| Build | fps | sim (systems) | view (gather) | copy | render |
|-------|----:|----:|----:|----:|----:|
| A3.1–A3.7 baseline | 25–30 | sim+read **23** (combined) | — | — | 11–12 |
| + A3.7a row-parallel levels | — | sim+read **18–19** | — | — | — |
| + A5.2a raw-memcpy gather (clean 4-way split) | **37** | **1.21** | 7.44 | 0.48 | 12.09 |
| + A5.2b parallel view refresh | — | 1.2 | **3.84** | 0.5 | ~12 |

**Headline:** simulating **1,000,000 agents = ~1.2 ms**. The gather is ~3.8 ms parallelised; the
~12 ms managed texture scatter is the ceiling (lever: GPU-instanced rendering — parked).

### 3b. `MassAIDemo` — HATE utility AI, 100,000 actors

Each actor is a `HateWorld` row; the whole population is scored/decided every frame as branchless
column passes, read back in one gather, scattered to a texture (the C3 harness). HUD split =
**dynamics** (bulk attribute Systems) / **decide** (the §8 AI) / **read** (gather + tally) /
**render** (swarm scatter).

| Model | fps | dynamics | decide (AI) | read | render |
|-------|----:|----:|----:|----:|----:|
| D5.1 single-ability (`EvaluateUtility`) | 53 | 5.5 | **0.50** | 1.75 | 5.83 |
| D5.1a (same, empty-effects early-out perf fix) | **79** | **0.16** | 0.46 | 1.65 | 5.7 |
| D5.2 multi-ability score→perturb→select (settling) | 75–80 | — | 1.2–1.4 | — | — |
| D5.2 (settled) | **85** | 0.20 | **0.95** | 1.28 | 4.52 |

**Headlines:**
- Deciding a single Heal ability for 100k actors ≈ **0.46 ms**; deciding between **two** abilities
  (two response-curve scoring passes + eligibility gates + two noise-perturb passes + argmax +
  Command override) ≈ **0.95 ms**. Adding the second ability + per-actor noise + argmax roughly
  doubled the AI cost and it is still under 1 ms.
- **D5.1a was the big win** (53 → 79 fps): the 5.5 ms `dynamics` was a managed interop artefact — an
  O(capacity)-not-O(live) per-row `IsValid` scan over the whole effects capacity (~400k rows) even
  with zero active effects. Early-out on live-count eliminated it. (Lesson for R&D: guard host-side
  per-row scans with a live-count early-out.)
- Same ceiling as the swarm: the frame is dominated by the single-threaded managed texture scatter,
  not the substrate. Engine + AI together ≈ 0.6 ms (D5.1a) / ~1.1 ms (D5.2).

> The D5.1 → D5.2 fps comparison is loose (different demo logic, lighter dynamics, simpler act/colour
> path). The comparable AI metric is the **`decide`** column; everything else is demo configuration.

## 4. Summary

The Phase-3 substrate is proven correct (114 cases, ASan+UBSan+TSan clean, deterministic parallelism)
and fast: the simulation cost of the full A→C→D vertical — DataLens Core → Unity binding → HATE — is
on the order of **~1 ms** at 1M-agent (pure sim) and 100k-actor (full utility AI) scale. The frame
budget is bound by visualisation, not the database. The one open correctness item (the serialise
round-trip xfail) belongs to the not-yet-started persistence/delta phase.
